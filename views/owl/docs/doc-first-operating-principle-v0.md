# Doc First Operating Principle

작성일: 2026-04-04
상태: v0

## 목적

Doc First 운영 원칙을 owl 관점에서 정의한다.
이 원칙은 단순히 문서를 많이 남기자는 뜻이 아니라, 중요한 판단·설계·작업·실행·결과를 문서 객체로 다루고 그 문서를 운영 계약과 증적으로 사용하는 방식이다.

## 핵심 정의

Doc First는 다음을 뜻한다.

- 중요한 생각과 결정을 채팅에서 휘발시키지 않고 문서로 승격한다.
- 실행 전에 목적, 범위, 입력, 출력, 제약, 완료 조건을 문서에 남긴다.
- 실행 후 결과, 변경점, 실패, 후속 조치를 다시 문서로 남긴다.
- 문서를 단순 기록이 아니라 사람과 에이전트가 함께 읽는 운영 인터페이스로 취급한다.

즉 Doc First는 문서가 계획의 시작점이고, 실행의 근거이며, 결과의 증적이 되는 운영 원칙이다.

## 왜 필요한가

채팅 기반 운영은 빠르지만 다음 문제가 있다.

- 중요한 판단이 대화 맥락 속에 묻힌다.
- 다음 세션이나 다른 런타임이 같은 맥락을 재구성하기 어렵다.
- 실행은 했지만 근거와 결과가 남지 않아 반복 실수가 생긴다.
- 장기 운영에서 구조, 정책, 용어가 흔들린다.

Doc First는 이 문제를 줄이기 위해 문서 생애주기를 운영 중심에 둔다.

## 문서 계층

Doc First는 문서가 모두 같은 역할을 하지 않는다고 본다.
각 계층은 다음처럼 구분한다.

### 설계/정책 문서
- 위치 예: `docs/`
- 역할: 개념, 설계, 계획, 스펙, 운영 규칙, 런북
- 질문: 왜 이렇게 하는가, 무엇을 기준으로 할 것인가

### 작업 문서
- 위치 예: `tasks/`, 프로젝트별 작업 문서
- 역할: 실제 실행 단위, 범위, 담당, 상태, 체크리스트
- 질문: 지금 무엇을 할 것인가

### 결과/증적 문서
- 위치 예: `reports/`, `compiled/*-report.md`
- 역할: 실행 결과, 진단, 검증, 감사, 관찰, 교훈
- 질문: 실제로 무엇이 일어났는가

### 장기 지식 문서
- 위치 예: `knowledge/`, `compiled/*-concept.md`, `compiled/*-index.md`
- 역할: 반복 참조되는 개념, 규칙, 구조, 맥락의 장기 보존
- 질문: 앞으로도 계속 참고할 기준은 무엇인가

### 기억/브리핑 문서
- 위치 예: `memory/`, `openclaw_report/`, `compiled/*-summary.md`
- 역할: 일시 맥락, 운영 요약, 빠른 전달, 장기기억 승격 전 단계
- 질문: 지금 맥락을 빠르게 어떻게 파악할 것인가

## 문서 생애주기

Doc First의 기본 흐름은 다음과 같다.

```text
아이디어/판단
-> 설계 또는 메모 문서화
-> 작업 단위 문서화
-> 실행
-> 결과/증적 문서화
-> 장기 지식 또는 기억으로 승격
```

이 흐름에서 중요한 점은 실행만 남기는 것이 아니라, 실행 전과 실행 후 모두 문서로 이어져야 한다는 것이다.

## 최소 문서 계약

새 작업이나 새 설계를 문서로 만들 때 최소한 다음 항목이 있으면 좋다.

- 목적
- 범위
- 입력
- 출력
- 제약
- 완료 조건
- 후속 조치

모든 문서가 이 형식을 강제할 필요는 없지만, 실행에 직접 연결되는 문서는 이 계약을 가능한 한 충족해야 한다.

## 운영 규칙

1. 중요한 구조 변경은 먼저 문서화한다.
2. 반복 작업은 문서에서 패턴을 식별한 뒤 승격 후보로 관리한다.
3. 실행이 끝나면 결과를 report, summary, note 중 하나 이상의 형태로 남긴다.
4. 장기 가치가 있는 결과는 concept, index, knowledge 문서로 승격한다.
5. 단순 채팅 답변만으로 끝나지 않게 하고, 재사용 가치가 있으면 파일로 남긴다.
6. 문서가 바뀌면 관련 README, 운영 규칙, 인덱스 문서도 함께 갱신한다.

## Agent CLI와의 관계

Doc First와 Agent CLI First는 경쟁 관계가 아니다.
둘의 관계는 다음처럼 본다.

- Doc First: 판단, 설계, 작업, 결과를 문서 계약과 증적으로 유지하는 상위 운영 원칙
- Agent CLI First: 그중 절차화 가능한 실행을 CLI 계약으로 외부화하는 실행 원칙

즉 흐름은 다음처럼 정리할 수 있다.

```text
문서화 -> 반복 패턴 식별 -> 스펙 정제 -> CLI 계약화 -> 실행 -> 결과 문서화
```

따라서 CLI First는 Doc First 위에서 더 강하게 작동한다.
문서 없는 CLI 승격은 맥락 손실을 낳기 쉽고, 문서만 있고 실행 계약이 없으면 반복성이 약하다.

## owl 편입 메모

owl 기준으로 보면 Doc First는 다음에 연결된다.

- `docs/agent-cli-operating-principle-summary-v0.md`
- `docs/term-dictionary-v0.md`
- `docs/rule-source-spec-v0.md`
- `docs/meta-governance-source-spec-v0.md`
- 향후 운영 원칙 인덱스, core concepts index, rule source index

또한 Brain 앱 데이터에서는 다음 유형으로 반영할 수 있다.

- summary: 운영 원칙 요약
- concept: 문서 중심 운영 개념 정리
- report: 실제 적용 사례와 검증 결과

## 적용 판단 메모

현재 Luca 워크스페이스에는 이미 다음 구조가 존재한다.

- `docs/` 설계와 계획
- `tasks/` 실행 단위
- `reports/` 결과와 감사
- `knowledge/` 장기 지식
- `memory/` 운영 기억
- `openclaw_report/` 경량 브리핑

즉 Doc First는 새 철학을 억지로 가져오는 것보다, 이미 존재하는 문서 기반 운영 습관을 상위 원칙으로 명문화하는 작업에 가깝다.

## 요약

Doc First의 핵심은 다음 한 문장으로 정리할 수 있다.

> 중요한 판단과 실행을 문서 객체로 다루고, 그 문서를 운영 계약과 증적으로 사용하는 것이 Doc First다.
