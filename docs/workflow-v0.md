# owl 워크플로 v0

작성일: 2026-04-03
상태: 설계 초안

## 1. 목적

owl의 워크플로는 원본 자료를 수집하고, LLM이 이를 구조화된 지식으로 컴파일하며, 필요할 경우 MDV를 통해 다양한 관점으로 다시 해석하고 활용하는 흐름을 정의한다.

핵심 목표는 다음과 같다.

- 원본 자료를 손실 없이 보존한다.
- LLM이 사람이 읽기 쉬운 지식 문서를 생성하게 한다.
- 질의응답과 리서치 과정에서 생성된 산출물을 다시 지식 자산으로 환류한다.
- 메타 관점은 원본을 대체하지 않고 보조한다.

---

## 2. 기본 흐름

### 단계 1. 수집
자료는 우선 `inbox/` 또는 `raw/`로 들어온다.

예시 자료:
- 웹 기사
- 논문 PDF
- GitHub 저장소 정보
- 이미지
- 대화 로그
- 메모
- 링크 목록

### 단계 2. 정리 전 보존
수집된 자료는 가능한 원형에 가깝게 `raw/`에 저장한다.

원칙:
- 원문 훼손 최소화
- 출처 정보 유지
- 저장 시점 기록 가능성 확보

### 단계 3. LLM 컴파일
LLM은 `raw/` 자료를 읽고 다음과 같은 `compiled/` 산출물을 만든다.

- 위키 문서
- 요약 노트
- 주제 문서
- 인덱스 문서
- 관계 설명 문서
- 보고서 초안

여기서 말하는 컴파일은 단순 변환이 아니라, 지식 구조화와 상호 연결을 포함한다.

### 단계 4. Surface / 열람 환경
사람 또는 에이전트는 raw, compiled, 파생 시각화와 산출물을 함께 볼 수 있는 surface를 사용한다.
현재 기본 surface는 Obsidian이다. raw 데이터, compiled wiki, Marp 슬라이드, matplotlib 이미지 같은 결과물을 모두 Obsidian에서 바로 열람하는 것을 기본 전제로 둔다.

### 단계 5. 열람과 질의응답
사람 또는 에이전트는 `compiled/`를 기반으로 탐색하고 질의한다.

출력 예:
- 답변 문서
- 슬라이드
- 요약 보고서
- 개념 비교 문서
- 시각 자료

위키가 어느 정도 커지면(대략 100개 문서, 40만 단어 규모를 가이드라인으로 삼음) fancy한 RAG 없이도 index, summary, concept 문서만으로 실용적인 Q&A가 가능하다는 가정을 운영 원칙으로 둔다.

### 단계 6. 환류
유의미한 산출물은 다시 `compiled/`나 `research/`로 환류한다.
이 filing loop 에 따라 owl은 점점 성장하는 지식 체계가 된다.

### 단계 7. 린팅과 헬스 체크
지식 베이스에 대해 다음과 같은 점검을 수행할 수 있다.

- 데이터 불일치 탐지
- 누락 항목 보완 후보 탐지
- 새로운 연결 후보 제안
- 새 문서 후보 제안

### 단계 8. 메타 해석
필요 시 MDV가 KIB를 읽고 다음을 수행한다.

- 차원 분류
- 관계 추출
- 프로파일별 뷰 생성
- 탐색 관점 제공

MDV는 초기에 필수 요소가 아니라 선택적 확장 요소로 둔다.

### 단계 9. 추가 도구 연결
필요 시 검색 도구, CLI 도구, 시각화 도구를 LKS에 연결해 더 큰 질의를 처리한다.

### 단계 10. 질의별 임시 위키와 반복 루프
큰 질문은 필요한 source를 모아 임시 위키를 구성하고, lint/health check를 거친 뒤, 반복 보강을 통해 최종 report를 만드는 흐름으로 확장할 수 있다.

---

## 3. 최소 운영 루프

```text
inbox -> raw -> compiled -> QA/output -> filing -> compiled
                    \-> MDV views
```

이 루프는 단순하지만, 원본 보존과 지식 성장이라는 두 축을 동시에 유지한다.

---

## 4. 역할 분리

### KIB가 맡는 것
- 자료 저장
- 정리 문서 보관
- 지식 열람 기반 제공

### MDV가 맡는 것
- 같은 자료를 여러 축으로 다시 보기
- 메타 수준 분류와 해석
- profile 기반 뷰 제공

### LKS가 맡는 것
- KIB와 MDV를 함께 포괄하는 전체 운영 체계
- Q&A, 리서치, 환류, 린팅까지 포함

---

## 5. 초기 구현 우선순위

1. inbox/raw/compiled 흐름 고정
2. compile 산출물 형식 고정
3. report/note/wiki 기본 템플릿 고정
4. 간단한 환류 루프 정립
5. 그 다음에 MDV 도입

---

## 6. 운영 원칙

- 원본은 지우지 않는다.
- 정리 문서는 덮어쓰지 말고 진화시킨다.
- 사람이 직접 편집하는 것보다 LLM 유지보수를 우선한다.
- 메타 뷰는 필요할 때만 강화한다.
- 설계는 가볍게 시작하고, 반복 패턴이 확인되면 승격한다.
- 초기 ingest는 완전 자율화하지 않고, source를 하나씩 직접 추가하면서 패턴을 고정한다.
- 패턴이 안정되면 `file this new doc to our wiki: (path)` 같은 한 줄 filing 동작으로 승격한다.
- 구조나 계약이 바뀌면 관련 문서를 즉시 갱신한다.
- 완료를 주장하기 전에 변경 반영 여부를 검증한다.
- 반복된 실수는 새 규칙이나 체크리스트로 흡수한다.
